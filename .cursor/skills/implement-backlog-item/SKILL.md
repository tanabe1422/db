# バックログ1件実施

**1セッション = BACKLOG 1件のみ。** 複数まとめて進めない。

ソース・オブ・トゥルース: [`docs/BACKLOG.md`](../../../docs/BACKLOG.md)

検証ルール: [`.cursor/rules/verify-after-change.mdc`](../../rules/verify-after-change.mdc) に従う（テストだけ通してビルド省略禁止）。

---

## ワークフロー（必須・この順）

進捗チェックリストをコピーして追跡する:

```
- [ ] 1. バックログ選ぶ
- [ ] 2. 説明＆プランたてる＆僕が承認
- [ ] 3. 実施
- [ ] 4. テスト（デグレがないか慎重に）
- [ ] 5. バックログ更新
```

### 1. バックログ選ぶ

1. `docs/BACKLOG.md` を読む。
2. 候補を決める（既定: 「全体の優先順位」の先頭、またはユーザー指定の `#`）。
3. ユーザーに候補を提示し、**1件に確定**する（曖昧なら確認する）。
4. 対象の難易度・推奨 AI・関連ファイルを短く把握する（必要ならコードを読む）。

### 2. 説明＆プランたてる＆僕が承認

1. 何をするか・触るファイル・受け入れ条件・検証コマンドを短く説明する。
2. **Plan モード**に切り替え（`SwitchMode` → `plan`）、`CreatePlan` でプランを出す。
3. **ユーザーがプランを承認するまで実装しない。** 拒否・修正指示があればプランを直して再承認を待つ。

プランに含めること:

- スコープ（この1件のみ）
- 変更予定ファイル
- 受け入れ条件
- 検証コマンド（関連テスト + ビルド）
- BACKLOG から消す／直す内容

### 3. 実施

1. 承認後に Agent モードで実装する。
2. スコープ外のリファクタ・隣接課題に手を出さない。
3. 既存の命名・パターンに合わせる。

### 4. テスト（デグレがないか慎重に）

変更領域に応じて **必ず** 実行し、失敗したら直してから次へ進む。

**frontend（`frontend/`）**

1. 関連テスト: `pnpm test -- <changed files or area>`
2. 型チェック＋ビルド: `pnpm build`（`tsc -b && vite build`）

テストファイルも `tsconfig.app.json` の include 対象なので、テストの型エラーもビルド失敗になる。

**Go（`*.go` / `internal/` など）**

1. 関連テスト: `go test ./...`（絞れるなら対象パッケージ）
2. ビルド: `go build ./...`

両方触ったら両方検証する。

**デグレ防止の追加観点**

- 変更に近い既存テストを優先して走らせる（「通るはずの周辺」を落とさない）。
- 挙動変更なら受け入れ条件をテストまたは手動確認手順で担保する。
- ドキュメント／story のみでも、frontend を触ったら `pnpm build` は省略しない。

### 5. バックログ更新

1. 完了した `#` を `docs/BACKLOG.md` から削除する。
2. 「全体の優先順位」とジャンル節・メモを現状に合わせて直す。
3. カバレッジ／ギャップ表があれば同期する。
4. ユーザー向けに、実施した `#`・要点・検証結果・次の候補を短く報告する。

---

## やってはいけないこと

- 承認前の実装
- 1セッションで複数 BACKLOG 項目を消化する
- テスト成功だけでビルドを省略する
- 完了したのに BACKLOG を更新し忘れる
- プランファイル自体をユーザー指示なく編集する（実施フェーズの成果物はコードと BACKLOG）
